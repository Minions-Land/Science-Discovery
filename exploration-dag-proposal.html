Proposal

Project-Level Exploration DAG for Autonomous Scientific Discovery

A cautious design for preserving the full research process, selecting evidence-backed paths, and preventing polished papers from outrunning their receipts.

DAG means directed acyclic graph. It does not mean RAG. RAG may enrich nodes, but the DAG is the durable project memory.
Executive Summary

Autonomous research needs a shared discovery graph that records questions, assumptions, hypotheses, experiments, citations, decisions, dead ends, and receipts. A final paper should be extracted from the highest-value supported path or subgraph, not invented after the fact as a polished narrative. This is a proposal, not a validated claim: its value must be tested through evidence quality, reproducibility, and audit speed.

Why Existing Systems Are Not Enough

The adjacent ecosystem already solves important slices of the problem. The missing layer is a project-native graph that can absorb those signals, keep failed branches alive, and later compress the best-supported subgraph into a paper or artifact.

Proposed System

The system is a persistent graph per project. Nodes represent scientific objects and workflow events. Edges represent dependency, support, contradiction, refinement, execution, citation, or decision relationships. Receipts bind nodes and edges to external evidence or local artifacts.

The system should support a tree-like view for humans and a DAG model for implementation. A strict tree is too weak because two branches can share the same dataset, citation, failed baseline, or methodological decision. A DAG lets knowledge converge without duplicating records. 

root question
  |-- Q-001: clarify the scientific question
  |-- A-002: expose a necessary assumption
  |-- H-003: propose a falsifiable hypothesis
  |     |-- E-008: run an experiment
  |     |-- C-014: attach a verified citation
  |     `-- D-017: promote, revise, or reject
  `-- DEAD-021: preserve a failed branch with its receipt

Data Model

A minimal graph needs typed nodes, typed edges, receipts, support status, and project boundaries. The schema should be simple enough for agents to append safely and strict enough for audits to reject unsupported promotion. 

project:
  id: ASD-001
  root_question: "Can hard-gated agent postures reduce unsupported scientific claims?"

nodes:
  - id: H-003
    type: hypothesis
    text: "A four-posture agent reduces premise and citation errors."
    support_status: tentative

  - id: E-008
    type: experiment
    command: "run benchmark against baseline agents"
    receipt: receipts/experiments/E-008.json

  - id: C-014
    type: citation
    claim: "ARA records dead ends in a trace layer."
    receipt: evidence/citations/C-014.json
    support_status: verified

edges:
  - from: H-003
    to: E-008
    relation: tested_by
  - from: C-014
    to: H-003
    relation: background_support

Node Types

question, assumption, hypothesis, experiment, result, citation, decision, and dead_end.

Support States

unverified, tentative, verified, refuted, blocked, and out_of_scope.

Workflow

  1. Branch: create a node for every meaningful question, assumption, hypothesis, experiment, citation, or decision.
  2. Record: attach actor, timestamp, prompt or command, input artifacts, and output artifacts.
  3. Verify: require receipts for citations, experiments, datasets, and major claims.
  4. Score: use decomposed criteria, not a single vague quality score.
  5. Promote: move only supported branches into the paper path.
  6. Retain: keep failed and negative branches with their reason for stopping.
  7. Extract: generate a paper or ARA-like artifact from the highest-value supported path or subgraph.

Minimal Implementation

The first version should avoid becoming a platform. It can be a file-backed artifact that extends the existing ARA style.

MVP behavior

Append-only team journal, YAML or JSON DAG, citation verification ledger, experiment receipt ledger, a simple graph view, and a path extraction command that exports a supported narrative skeleton.

Evaluation Plan

The proposal should be tested against ordinary multi-agent chat, ARA-only artifact packaging, and run-tracker-only workflows. It should be revised or rejected if it creates impressive diagrams without improving evidence quality.

Evaluation questionPossible metric
Does the DAG reduce unsupported claims?Unsupported-claim rate before and after graph gates.
Does it prevent repeated failed work?Number of repeated dead-end attempts per project.
Does it improve audit speed?Time for a new agent or human to verify a claim path.
Does it help paper writing?Fraction of final paper claims linked to verified nodes.
Does it preserve novelty?Path overlap between extracted paper candidates.
Does it stay usable?Duplicate-node rate, graph size, retrieval success, user correction rate.

Non-Goals and Risks

Non-Goals

  • Not a replacement for human scientific judgment.
  • Not a claim that every research process is a clean tree.
  • Not a general RAG product.
  • Not a proof engine that turns verifier scores into truth.

Risks

  • The graph may become a log archive that nobody uses.
  • Agents may optimize for well-formed nodes instead of better science.
  • Weak receipt schemas may create false confidence.
  • Cross-project reuse may spread mistakes if provenance is shallow.

Closing

The strongest version of this proposal is not a prettier research dashboard. It is a discipline for zero-hallucination scientific work. Agents may explore broadly, but promotion into a paper requires evidence. The DAG is the shared memory that lets the team see not only the final path, but also the abandoned branches, missing receipts, and negative results that made the final path credible.
