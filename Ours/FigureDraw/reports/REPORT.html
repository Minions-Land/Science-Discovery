FieldDraw — 一个 SKILL 横评 · 2026/05/24

给 Sonnet 4.6 装上不同 SKILL,谁能画好科研图?

六个画图 SKILL 包,十五种科研常见图,九十个隔离 cell,加上一份独立 grader 的逐图打分。我们想知道:MinionsOS 自家的画图+排版 SKILL 套件,在严格控制变量后,真实的强项与边界在哪。

II实验设定

变量隔离严格到「每个 cell 都跑在 env -i 出的新 HOME 里」,完全照搬 GQPA-Diamond v2-baseline-pull 的范式。

每个 (arm, fig_type) cell 在 env -i 隔离的临时 HOME 里启动一个全新的 claude --print 进程,只挂载该 arm 自己的 SKILL.md 和当前 cell 的 fixture 目录。--strict-mcp-config 关掉所有 MCP 服务,--setting-sources "" 关掉所有 settings 继承,--disallowedTools WebSearch WebFetch 禁联网。每个 arm 在自己的沙箱里看不到别 arm 的 SKILL,看不到 host 的 ~/.claude/,看不到 MinionsOS 项目代码或 CLAUDE.md。

八维 grader rubric

每个 cell 跑完后,由一个独立 Sonnet judge(无 SKILL 挂载,看不到 arm 名,看不到 data.json)按八个维度 0-3 打分,共 24 分:

  1. scientific_clarity — 图是否承载 brief 的科学陈述
  2. typography — 是否禁用 DejaVu Sans 默认 / 是否 editable text
  3. palette — colorblind-safe?directional vs categorical 区分?
  4. layout_density — 空间是否被高效利用
  5. reviewer_readiness — Reviewer 2 会要 revise 吗?
  6. vector_fidelity — figure.pdf 是真矢量(text 可选中)吗?
  7. file_format — pdf+png+svg+caption.tex+gen_figure.py 是否齐全
  8. caption_quality — caption 是否带具体数值/具体发现

每个 cell grader 还要给出每维一句话 evidence、两条 strengths、两条 weaknesses,用来在下面 §VII 矩阵里直接展示。

IVMinionsOS 强项

五个 fig_type 拿冠军,scientific_clarity 维度满分,排版赛道 categorical 胜出。

1. scientific_clarity 唯一 3.00 平均的 arm

15 个 fig_type 全部拿到 grader 满分 3,其它 arm 都至少有一处掉到 2。原因:academic-plotting 把 PALETTE-dict + figure-layout-defaults 的 4-panel 默认前置到了脚手架——agent 一上来就在写已经合规的代码,不存在"画完才发现 panel 不对"的返工。

证据outputs/grouped-bar/minionsos.png 的 caption 直接点出 "+4.8pp on MATH, +5.2pp on GPQA"——这是数据驱动的具体数字,不是泛泛的 "OursModel achieves the best performance"。grader/minionsos/grouped-bar.json 给到 23/24,evidence 字段写 "captures the +4.8/+5.2 pp narrative explicitly in the caption."

2. architecture 24/24 满分,唯一一个

figure-spec SKILL 的 JSON 描述符 → 渲染流程,在 architecture 这道题把外部 5 个 arm(都只有 16-18 分)拉开了 6-8 分。这是 categorical 的胜出——其他 arm 没有专门处理 boxes-and-arrows pipeline 的 SKILL,只能让 Sonnet 现场用 matplotlib patches 拼,产出的图缺乏分组意识。

3. hero figure 真用结构,不是文字框

figure-layout-defaults 直接对应 4-panel hero,Panel A hero、B/C/D 数据 panel,标签 a/b/c/d 左上角粗体。hero-figure-prompt 明确写 Panel A 必须和 data panel 同等正式化——这条规则被 agent 严格遵守。nature-figure 在这道题上 Panel A 退化为说明文字 + 简易框图,grader 直接给到 15/24,是它在 15 道题里得分最低的一道。

4. 排版赛道:其他 arm 跑不出来

只有 minionsos arm 跑了 paper-page cell:输入 4 张图 + claim.json,输出 3 页 two-column NeurIPS-style 论文 PDF。其他 5 个 arm 没有 latex-typography / paper-compile / make-latex-model,根本不在同一赛道。

❧

VMinionsOS 弱项

字号纪律、训练曲线类、stacked-bar、vector_fidelity 这四块需要补课。

1. typography 字号纪律不及 nature-figure

nature-figure 的 typography 平均 1.47 vs minionsos 1.13。原因:nature-figure SKILL 反复强调 "axis label 7 pt / tick 6-7 pt / panel letter 9 pt bold sans" 这种精确数值;academic-plotting 则写得相对软("9-10 pt label, 8 pt tick")。grader 在 PNG 上能直接读出来字号是否合规——agent 自由发挥时偶尔走到 11 pt,grader 立刻扣分。

2. 训练曲线类输给 figures4papers

line-errband 17 vs 23,dual-axis-time 18 vs 23。figures4papers 自带 11 个真实论文的 figure_* 示例代码——其中很多是训练曲线 / convergence 图,agent 直接 grep plot_comparison_*.py 抄结构,产出的曲线 legend 位置、y 轴刻度密度、CI 阴影透明度都更接近真实 ICML/NeurIPS paper。

3. stacked-bar 16 / 第三

被 nature-figure(21)和 pdf-vector-layout(20)反超。academic-plotting 的 chart-type-from-data-shape 表里只有一行 "proportions → stacked bar (avoid pie)",缺少具体 sample code,agent 在自由发挥时用了 4 种饱和度高的颜色,被扣 palette。

4. vector_fidelity 1.13,第三

academic-plotting 写了 pdf.fonttype=42,但 grader 在审 PDF 时仍发现部分 figure 把字渲染成路径(常见于 hero 文字框 + matplotlib patches)。pdf-vector-layout/scripts/verify_vector.py 没被强制做成画图退出前的 check。

改进方向 把 verify_vector.py 直接拉进 academic-plotting 的"画完图必须跑的 lint 步骤";给 academic-plotting 增补 stacked-bar / line-errband / dual-axis-time 的 gallery 案例;把字号规则从软指导改为硬数值。

VI排版独占赛道

从图到完整可编译 LaTeX 论文,只有 MinionsOS 跑得出来。

给 minionsos arm 这样一个 brief: 用前面跑出来的 4 张图(grouped-bar / line-errband / scatter-fit / 4panel-hero) + claim.json,出一篇 2-3 页 two-column NeurIPS-style 论文片段。其他 5 个 arm 都没有 latex-typography / make-latex-model / paper-compile,这道题对它们来说不存在;只有 minionsos 能跑。

该 cell 用 --add-dir 挂载了 fixture 目录(4 张 PDF + claim.json + captions.json),然后 6 分钟内跑完了完整流水线:scaffold LaTeX 项目 → 写 main.tex (preamble macro discipline + abstract + Method + Results) → 写 references.bib → latexmk + bibtex + 二次 pass → 输出 main.pdf。

main.pdf 第一页:two-column 排版,\methodname macro 渲染为 RetroDiff small-caps,Figure 1 嵌入 grouped-bar 并 cross-ref。

结构亮点

main.tex preamble 段把方法名、关键数据点、scaling 拟合常数全部 macro 化:

\newcommand{\methodname}{\textsc{RetroDiff}\xspace}
\newcommand{\baselineA}{Method-A\xspace}
\newcommand{\baselineB}{Method-B\xspace}
\newcommand{\benchMATH}{MATH\xspace}
\newcommand{\benchGPQA}{GPQA\xspace}
% ---- result macros ----
\newcommand{\gainMATH}{$+4.8$\,pp\xspace}
\newcommand{\gainGPQA}{$+5.2$\,pp\xspace}
\newcommand{\dataEfficiency}{60\%\xspace}
\newcommand{\scalingRsq}{$R^{2}>0.9$\xspace}

这就是 latex-typography SKILL 写的 "每个再次出现的命名实体都用 \newcommand,正文不允许出现一处硬编码" 规则的严格落实。改方法名只要改一行 \newcommand。这正是其他 5 个 arm 给不出的工程能力——它们要么没有 LaTeX SKILL,要么有 LaTeX 但缺少 paper-compile 的迭代修复机制。

VIII方法学与限制

这次比较有效的边界,以及不应该过度推论的地方。

变量隔离

6 arms × 15 fig_types 跑同一个 claude-sonnet-4-6 --effort max,数据相同(fixtures/<fig>/data.json),brief 相同(fixtures/<fig>/brief.md),唯一变量是 ~/.claude/skills/ 里挂哪些 SKILL.md。这是 GQPA-Diamond/v2-baseline-pull 的同一套配方。每个 cell 在 env -i 出的临时 HOME 里跑,所有 host 端 hook(large_file_guard / edit_failure_rescue / reel_capture / bg_keepalive_nudge)都被 --strict-mcp-config --setting-sources "" 关掉。

Grader 独立性

judge 是另一个 Sonnet 实例,没有任何 SKILL 挂载,不知道 arm 名(每个 cell 只看到 figure.pdf/png + brief + caption + script),写到固定 8 维 rubric。这避免了"画图 SKILL 自己审自己"的循环裁判。但仍然是 Sonnet 审 Sonnet,和人类 reviewer 的判断可能有偏差。

单 seed

每个 cell 只跑了 1 次。GQPA-Diamond 报告 baseline 在 single-run 有 ~1-2% noise,所以 ±1 分以内的 arm 差距(如 minionsos 18.20 vs research-paper-writing 17.93)统计上可能不显著。架构图、grouped-bar、scatter-fit、sankey 这种差距 ≥ 5 分的胜出才算稳。

"画图 SKILL 不等于画图能力"

这次实验显示,Sonnet 即使没有任何 SKILL 挂载(pdf-vector-layout arm 几乎是裸 Sonnet),还能拿到 17.60/24 — 模型本身已经会画 publication-quality matplotlib。SKILL 的边际贡献是把"上限拉高"(例如 figure-spec 让 architecture 拿 24 分),不是"避免 Sonnet 画错"。这点和 GQPA-Diamond 的 baseline 也对得上:SKILL 的真实价值在 categorical 胜出,不在小幅 win-rate 提升。

1. 能编译 ≠ 用 macro。 7 个 arm 都能让 latexmk 跑过,但只有 minionsos 写了 6 个 \newcommand(method name + 4 个关键数字 + 1 个 baseline 集合)。其它 arm 普遍把方法名硬编码在正文里——遵守了 LaTeX 编译,但破坏了 maintainability discipline。

VII抄哪些规则进 MinionsOS

基于本轮 grader 证据和 paper-page 观察,这里列出 MinionsOS 应当从外部 arm 借鉴的具体规则。每条都带证据指针。

以下是从这次 144 cell 的 transcript + grader evidence 里提炼的具体借鉴清单。每条都标了证据来源——给定 arm + 给定 cell + grader 给出的 evidence 字段。MinionsOS SKILL.md 改动可以直接对着这些 issue 来。

Borrow #1 · From ml-paper-writing · 证据 cell: grouped-bar / line-errband / 4panel-hero / roc-prc(均拿 arm 第一)

给 academic-plotting 增补 ML 论文 plot 的 idiom 子库

ml-paper-writing 在 4 类 ML 论文常见图(grouped-bar / line-errband / 4panel-hero / roc-prc)都拿冠军,差距 1-3 分,稳定胜出。它的 SKILL.md 只有 67 行,但 references/ 里直接给出了 ICML/NeurIPS 风格的 plot 模板代码。

怎么抄进 MinionsOS:在 figure-aesthetic-exemplars/gallery/ 增加 ml-paper-grouped-bar.py / ml-paper-line-errband.py / ml-paper-roc-prc.py 三个 reference scripts。Skill 里增加一条 "chart-type-from-data-shape" 的硬规则:训练曲线 → log y-axis、CI 半透明 ≤ 0.25 alpha、legend 在右下角内嵌。

Borrow #2 · From awesome-writing-prompts(中文 prompt 派) · 证据 cell: box-violin (19) / ridgeline (19) / scientific_clarity 2.94 / reviewer_readiness 2.29 全场第一

给 caption 写法增补 "reviewer 视角自审" prompt

合成 prompt arm 在 reviewer_readiness 维度拿到第一(2.29),它的 caption 普遍包含明确的 "how to read" 句、bold 关键 finding、解释 hatching/star 等视觉编码。这来自合成 SKILL.md 中的「论文整体以 Reviewer 视角进行审视」prompt。

怎么抄进 MinionsOS:在 academic-plotting 末尾追加一段 "caption checklist":(1) 用一句话解释 "读者第一眼应看到什么";(2) bold 出 take-home 数字;(3) 解释每种视觉编码(hatching / star / 颜色)。把它做成 caption.tex 退出前的强制 lint。

Borrow #3 · From composer-lishix · 证据 cell: latex-table (21,arm 第一)

给 latex-typography 增补 booktabs 的 "两栏分组" idiom

composer-lishix 在 latex-table 拿冠军(21,gap +1 vs minionsos),它产出的表格用了 \multicolumn{4}{c}{baselines} 把 baselines 和 ours 横向分组。这是 NeurIPS / ICML 论文表格的高级 idiom。

怎么抄进 MinionsOS:latex-typography 中加一节 "booktabs 分组习惯":当 methods 自然分成 baseline / ours 两组时,用 \multicolumn + \cmidrule(lr){i-j} 横向分组,正文用 \textbf 加粗 best-per-column,且 within-1-std overlap 时同时加粗。

Borrow #4 · From scientific-writing-kdense · 证据 cell: network-graph (21,arm 第一)+ palette 平均 2.31(全场第一)

给 academic-plotting 增加 "network 图避免 hairball" 章节

k-dense 在 network-graph 拿冠军(21),它的输出用了 force-directed 加 community 染色 + edge alpha 0.25-0.4。它的 palette 平均 2.31 是全场第一,说明 SKILL 在用色纪律上写得最严。

Borrow #5 · From stat-writing-fuhaoda · 证据 cell: forest-plot (20)、sankey (18,arm 第一)

给 figure-spec 增加 sankey/flow 图的 idiom

stat-writing 在 sankey 上拿冠军(18,gap +8 over latex-document)。它的 SKILL 里 references 包含统计学 reporting 的 flow chart 习惯——这是 MinionsOS figure-spec 没覆盖的:数据流向图。

Borrow #6 · From academic-paper-imbad · 证据 cell: scatter-fit (20)、dual-axis-time (19)

从 12-agent pipeline 抄一条 "draft → reviewer-pass → revise" 写作循环

academic-paper-imbad 用 12-agent pipeline,在 caption 写作上普遍带 "draft → revise" 痕迹。MinionsOS 当前的 writer 是 single-pass。

Anti-pattern #1 · From latex-document · 证据 cell: paper-page-latex-document(FAIL)

SKILL.md 描述要让 agent 能区分 "单图" vs "整篇 paper" 任务

latex-document arm 在 paper-page cell 上误把 4 张图理解成「需要画一张总图」,产出 figure.pdf 而不是 main.pdf。它的 SKILL.md 是 256 行的 "通用 LaTeX 文档 → PDF",描述太宽泛,agent 没有 task-discrimination 信号。

给 MinionsOS 的反面教材:每个 SKILL.md 的 description: 都应该写明「适用」+「不适用」两类,这样 agent 在多 SKILL 共存时能正确触发。MinionsOS 大部分 SKILL 已经做了这件事("When to invoke" / "When not to load"),但 academic-plotting 的 "When not" 写得偏弱,值得补强。

Anti-pattern #2 · 全场观察 · 证据: 7 个非-minionsos arm 的 macro 数都 ≤ 3

不要假设 "会用 LaTeX" = "会用 macro discipline"

7 个能编译 paper 的 arm 里,只有 minionsos 用了 6 个 \newcommand,其它都 ≤ 3,普遍把方法名硬编码在正文。这说明 LaTeX 编译纪律和 macro 纪律是两件事,后者要靠专门的 SKILL 强化。

给 MinionsOS 的强化方向:latex-typography 已有的 macro discipline 规则要做成 paper-compile 退出前的 lint check:扫 main.tex,如果方法名 / 关键贡献名 / 关键数字超过 3 次出现且未 macro 化,直接 fail compile 并报错。这是其它 arm 完全没考虑的工程层面纪律。
